The user wants to build a lightweight text/diary interface for smell reports that is report-first rather than map-first. Let me understand the flow:

User provides their zip code and a reference intersection to define their approximate lat/lng
The app fetches smell reports that contain personal/sentiment language (personal pronouns, breathing/air words, beyond just "industrial" "stink" "rotten eggs")
Reports are sorted by length (text depth)
User can select a report to see details including distance from their location



Plan: smellPGH report-first diary interface
Context
Building a lightweight, report-first text interface for Smell PGH data — the inverse of the existing map. The goal is human connection: letting people read other people's air quality experiences, filtered to reports with personal/experiential language, sorted by depth. Lives in A2_CAPTCHAs/c3/ alongside META.md.

Output
Single file: A2_CAPTCHAs/c3/index.html
Vanilla JS, no build step, opens directly in browser (file:// or local server).

Data flow
User sets location — enters zip + optional cross-street. Nominatim geocodes it to lat/lng. Stored in JS memory only (no cookies, no localStorage).

Fetch reports — GET https://api.smellpittsburgh.org/api/v2/smell_reports?region_ids=1&start_time=X where X = start of current week (Monday 00:00 local). No auth needed.

Filter for sentiment — keep reports whose combined text matches at least one word from:

Personal pronouns: i, my, me, we, our, us, you, your
Body/air: breath, breathing, breathe, lung, nose, throat, cough, wheeze, asthma, chest, eye, skin
Sensation/place: headache, nausea, dizzy, sick, burning, stinging, outside, inside, window, door, house, home
Regex: /\b(i|my|me|we|our|us|you|your|breath|breathing|breathe|lung|nose|throat|cough|wheeze|asthma|chest|headache|nausea|dizzy|sick|burning|outside|inside|window|door|house|home)\b/i
Applied to concatenation of all three text fields.
Sort — descending by total text length (smell_description + feelings_symptoms + additional_comments).

Sort — primary: most recent first (observed_at descending). Secondary tiebreak within same timestamp: longer text first.

Render list — each item shows:

Index number + smell_value (1–5 dots or ●●●○○ style)
Zip code + formatted timestamp
~80-char preview snippet from combined text
Select a report — click expands it inline showing:

Full text (all three fields, labeled)
Distance from user's location (haversine, in miles, to one decimal)
Report's lat/lng (small, for reference)
Key implementation details
Nominatim geocode call:


GET https://nominatim.openstreetmap.org/search?q={zip}+{cross_streets}+Pittsburgh+PA&format=json&limit=1
Header: User-Agent: smellpgh-diary/1.0
Falls back to zip-only if cross-street is blank.

Haversine (JS, inline):
Standard formula — returns miles. Applied only when a report is expanded.

Timestamp display:
new Date(observed_at * 1000).toLocaleString('en-US', {month:'short', day:'numeric', year:'numeric', hour:'numeric', minute:'2-digit'})

Smell value display:
'●'.repeat(smell_value) + '○'.repeat(5 - smell_value)

API CORS: api.smellpittsburgh.org returns Access-Control-Allow-Origin: * — browser fetch works directly.

No date-range UI for now — pulls this week (Mon 00:00 to now). Can add later.
